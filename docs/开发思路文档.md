## 三、M1 — 登录认证

> **SQL 文件**：[sql/m1_login.sql](sql/m1_login.sql)

### 3.1 逻辑

```
输入：ref_id（医生ID / 病案号）、密码、角色（医生/病人/管理员）
输出：验证通过 → 进入对应角色主页；失败 → 提示错误
```

### 3.2 要点

- Account 表走 **多态引用**：`ref_id` + `role` 决定指向 `doctor` 还是 `patient`。
- 登录成功后 session 存储 `(account_id, ref_id, role)`，后续所有操作从 session 取角色做权限校验。
- 管理员账号的 `ref_id` 无对应实体表，仅为标识。
- 密码哈希比对在应用层完成，SQL 只取出 `password_hash`。

## 四、M2 — 管理员基础数据维护

> **SQL 文件**：[sql/m2_admin_maintenance.sql](sql/m2_admin_maintenance.sql)

### 4.1 科室管理

标准 CRUD。**删除科室必须先检查**：该科室下是否有医生？是否有病房？两项均为 0 才可删除。SQL 文件已含两条前置检查。

### 4.2 医生管理

**新增医生必须在一个事务中同时创建 account 记录**：

```
事务：
  1. INSERT INTO doctor → 获取 doctor_id
  2. INSERT INTO account (ref_id=doctor_id, role='医生', password_hash)
```

**删除医生必须检查 3 项**：
1. 未来日期的排班（schedule）
2. 待就诊的挂号（registration）
3. 仍有在院病人的住院档案（admission_file WHERE discharge_time IS NULL）

三项均为 0 才可删除。

### 4.3 药品管理

标准 CRUD。唯一注意：修改药品单价**不会影响已生成处方**的费用——历史处方的费用在查询时实时 JOIN `medicine.unit_price` 计算。如果需求方要求历史费用锁定，需要改为在 `prescription_detail` 中冗余存储下单时的单价。

## 五、M3 — 排班管理

> **SQL 文件**：[sql/m3_schedule.sql](sql/m3_schedule.sql)

### 5.1 创建排班的 3 层检查

创建排班是整个系统 **校验逻辑最密集** 的操作，必须在插入前执行：

```
输入：doctor_id, sched_date, start_time, end_time, sched_type, clinic

检查1（R05）：sched_type = '门诊坐诊' → clinic 必填
               sched_type = '住院巡诊' → clinic 留 NULL

检查2（R03 + R06）：同一医生同日所有排班（不分类型）不得时间重叠

检查3（R04）：同诊室同时段不得排两位医生（仅 sched_type = '门诊坐诊' 时执行）
```

### 5.2 时间段重叠判定

两条排班 A、B 重叠的充要条件：

```
A.start_time < B.end_time AND A.end_time > B.start_time
```

SQL 文件中 `4.2` 节两条 SELECT 即基于此公式。编辑排班时逻辑相同，只需在 WHERE 中排除自身（`AND schedule_id != ?`）。

### 5.3 要点

- 同一医生的门诊和巡诊之间的互斥（R06）已由检查2 覆盖（不区分 sched_type）。
- 排班列表查询已 JOIN doctor + department，前端可直接渲染。

## 六、M4 — 病人挂号

> **SQL 文件**：[sql/m4_registration.sql](sql/m4_registration.sql)

### 6.1 挂号流程

```
步骤1：病人选择科室 → 系统查询该科室当日有门诊坐诊排班的医生（SQL 5.1）
步骤2：病人选择医生/时段 → 系统判定初诊/复诊（SQL 5.2）
步骤3：创建挂号记录（SQL 5.3），reg_status 固定为 '待就诊'
```

### 6.2 初诊/复诊判定

查询该 `case_no` 下是否有 `reg_status = '已就诊'` 的历史记录。有 → 复诊，无 → 初诊。

### 6.3 挂号容量限制（R08）

当前 `schedule` 表没有 `max_registrations` 字段。如需实现容量控制，需给 schedule 加字段，然后在挂号前 COUNT 当前已挂号数。SQL 文件 `5.6` 已预留注释。

## 七、M5 — 医生门诊接诊与处方

> **SQL 文件**：[sql/m5_outpatient_consultation.sql](sql/m5_outpatient_consultation.sql)

### 7.1 接诊流程

```
步骤1：医生查看待就诊队列（SQL 6.1）→ 选择病人接诊
步骤2：更新挂号状态为 '已就诊'（SQL 6.2）
步骤3：查询可选药品（库存 > 0，R12）（SQL 6.3）
步骤4：开具处方 + 明细（SQL 6.4，事务）
```

### 7.2 开具处方的事务

这是一个事务，含 3 步 SQL（见 `6.4`）：

1. `INSERT INTO outpatient_prescription` → 获取 `presc_id`
2. 逐条 `INSERT INTO prescription_detail`，`presc_type` 固定为 `'门诊'`
3. （可选）`UPDATE medicine SET stock_qty = stock_qty - ?`

任何一步失败必须回滚。费用不存储在 outpatient_prescription 中，查询时实时 JOIN 计算。

### 7.3 费用计算（R14）

```
诊疗费 = doctor.consultation_fee
药费   = Σ (prescription_detail.qty × medicine.unit_price)
总费用 = 诊疗费 + 药费
```

SQL 文件 `6.6` 是费用汇总查询，一次 JOIN 算出诊疗费 + 药费总和。

### 7.4 多态处方明细

`prescription_detail` 通过 `presc_type` 枚举（'门诊'/'住院'）区分类型。`presc_id` 在 '门诊' 时指向 `outpatient_prescription`，在 '住院' 时指向 `inpatient_prescription`。数据库层面**无 FK 约束**，应用层必须保证：
- 每条 INSERT 同时填 `presc_id` + `presc_type`
- 每条 SELECT 必须带 `presc_type` 条件
- 删除门诊处方时，需同步删除关联的明细（应用层级联）

## 八、M6 — 病人缴费与查看

> **SQL 文件**：[sql/m6_patient_payment.sql](sql/m6_patient_payment.sql)

### 8.1 缴费流程

```
步骤1：查看待支付处方列表（SQL 7.1）→ 点击某张处方
步骤2：查看费用明细 + 汇总（SQL 7.2 + 复用 M5 的 6.5）
步骤3：确认支付（SQL 7.3）
```

### 8.2 支付操作

`UPDATE outpatient_prescription SET pay_status = '已支付'`，带 `WHERE pay_status = '待支付'` 防重复支付。如果受影响行数 = 0，说明已被支付或不存在，前端需提示错误。

### 8.3 病人历史全景查询（R31）

分两条 SQL 独立执行：
- **门诊历史**（SQL 7.4）：以 `case_no` 查 registration → outpatient_prescription
- **住院历史**（SQL 7.5）：以 `case_no` 查 admission_file → ward/bed
- **住院费用明细**（SQL 7.6）：按 `file_no` 查每日记录 + 处方 + 药费明细

前端可将两个结果集合并展示为统一时间线。

## 九、M7 — 住院登记（入院）

> **SQL 文件**：[sql/m7_admission.sql](sql/m7_admission.sql)

### 9.1 入院流程

```
步骤1：选择科室 → 查可选病房（有空床、科室对口）（SQL 8.1）
步骤2：选定病房 → 查该病房空闲床位（SQL 8.2）
步骤3：缴纳预交金 → 执行入院事务（SQL 8.3）
```

### 9.2 入院事务（6 表联动，必须原子执行）

```
事务内 5 步操作（SQL 8.3）：

  A. SELECT ... FOR UPDATE 锁床位 — 防并发
  B. INSERT INTO admission_file — 创建档案（admit_time 自动填入，discharge_time = NULL）
  C. UPDATE patient — 标记 is_inpatient='是'，填入 ward_id/bed_id
  D. UPDATE bed — 标记 bed_status='占用'
  E. UPDATE ward — occupied_cnt + 1
```

任何一步失败 → 全部回滚。`FOR UPDATE` 是并发控制的关键：两个管理员同时分配同一床位时，后到者会被阻塞直到前者提交，然后发现 `bed_status != '空闲'` 从而回滚。

### 9.3 要点

- `deposit_balance` 初始值由业务传入。
- `admit_time` 由数据库 `DEFAULT CURRENT_TIMESTAMP` 自动填入。
- 入院后 patient 表的 `ward_id/bed_id` 是冗余字段，与 admission_file 保持一致（R20）。

## 十、M8 — 医生每日查房

> **SQL 文件**：[sql/m8_daily_rounds.sql](sql/m8_daily_rounds.sql)

### 10.1 查房流程

```
步骤1：查看我的在院病人列表（SQL 9.1）
步骤2：选择病人 → 填写治疗方案描述 → 开药 → 提交（SQL 9.2，事务）
```

### 10.2 每日查房事务（8 步，必须原子执行）

```
事务内（SQL 9.2）：

  A. 检查当日是否已有记录（R21）→ 有则拒绝
  B. SELECT ... FOR UPDATE 锁档案，获取 deposit_balance + charge_rate
     （余额 ≤ 0 则拒绝 — R24）
  C. INSERT INTO inpatient_record (daily_fee 初始为 0)
  D. INSERT INTO inpatient_prescription
  E. 逐条 INSERT INTO prescription_detail (presc_type = '住院')
     （一张处方含多条用药明细，循环执行）
  F. 聚合当日药费 — 通过 inpatient_prescription JOIN prescription_detail
     按 record_id 汇总当日所有用药（这是与原文档的关键修正）
  G. UPDATE inpatient_record SET daily_fee = 床位费 + 药费
  H. UPDATE admission_file SET deposit_balance = deposit_balance - daily_fee
     （WHERE deposit_balance >= daily_fee 兜底，受影响行数 = 0 则回滚）
```

### 10.3 要点

- 一天最多一条住院记录（R21），重复提交直接拒绝。
- 住院记录 ≠ 住院档案：档案入院时建一次，记录每天建一条。一个档案下挂多条记录。
- 药费聚合时必须 `JOIN inpatient_prescription` 而不能直接用 `presc_id` 查 `prescription_detail`——因为 `prescription_detail` 的 `presc_id` 需要配合 `presc_type = '住院'` 才能唯一确定住院处方记录，JOIN 方式是最安全的取法。

## 十一、M9 — 出院结算

> **SQL 文件**：[sql/m9_discharge.sql](sql/m9_discharge.sql)

### 11.1 出院事务

```
事务内（SQL 10.1）：

  A. SELECT ... FOR UPDATE 锁档案 → 获取 case_no, ward_id, bed_id
  B. 校验 NOW() > admit_time（R25）
  C. SELECT SUM(daily_fee) AS total_cost（R26）
  D. 计算 结算金额 = deposit_balance - total_cost（R27，应用层计算）
  E. UPDATE admission_file SET discharge_time = NOW()
  F. UPDATE patient SET is_inpatient='否', ward_id=NULL, bed_id=NULL
  G. UPDATE bed SET bed_status='空闲'
  H. UPDATE ward SET occupied_cnt = occupied_cnt - 1
```

### 11.2 要点

- 出院后 `admission_file.discharge_time` 有值，`patient` 的冗余字段清空。
- 结算金额为正 → 退还，为负 → 补缴。
- 所有 `inpatient_record` 保留不删，可随时回溯。
- 出院后查历史费用用 `10.2`，可显示住院天数（`DATEDIFF`）。

## 十三、需要应用层保证的约束汇总

| 编号 | 约束 | 实现方式 |
|------|------|----------|
| R02 | Account.ref_id 按 role 路由 | 多态引用，应用层路由 |
| R03 | 医生同日排班不重叠 | 插入前 SELECT 检测（SQL 文件 M3 4.2） |
| R04 | 同诊室同时段不冲突 | 同上（仅门诊坐诊时检测） |
| R05 | 门诊排班诊室必填 | 应用层表单校验 |
| R08 | 挂号容量限制 | schedule 表尚无容量字段，预留扩展 |
| R12 | 仅展示库存 > 0 药品 | WHERE 条件即可 |
| R16-R17 | 病房/床位有空才分配 | WHERE 条件即可 |
| R21 | 每日最多一条住院记录 | 插入前 COUNT 检查（SQL 文件 M8 9.2 Step A） |
| R24 | 预交金不足次日停医疗 | 查房前检查余额（SQL 文件 M8 9.2 Step B） |
| R25 | 出院时间 > 入院时间 | 应用层判断 |
| — | PrescriptionDetail 引用完整性 | 多态设计无 FK，应用层级联删除 |

## 十五、通用开发规范

1. **事务边界**：文档中标注了"事务"的模块（M5/M7/M8/M9），任何一步失败必须全部回滚。
2. **并发控制**：入院分配床位、查房扣减余额时用 `SELECT ... FOR UPDATE`。
3. **费用计算**：总费用每次 JOIN 实时计算，不冗余存储。药品单价变动不影响历史处方费用。
4. **多态引用**：`prescription_detail` 的 INSERT 要同时填 `presc_id` + `presc_type`，SELECT 必须带 `presc_type`。
5. **冗余字段**：`patient.ward_id/bed_id` 的权威来源是 `admission_file WHERE discharge_time IS NULL`，冗余字段仅用于快速查询，逻辑判断以 admission_file 为准。
