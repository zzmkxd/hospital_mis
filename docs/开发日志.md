## 2. 架构决策

### 2.1 三层分离：SQL 文件 → 路由代码 → 开发文档

- **`sql/*.sql`** — 纯 SQL，可复制到数据库工具直接执行验证
- **`routers/*.py`** — 将 SQL 嵌入路由函数，处理 HTTP 请求/响应
- **`开发思路文档.md`** — 业务逻辑说明，引用 SQL 文件路径但不嵌入 SQL

**原则**：SQL 文件是单一事实来源。路由中的 SQL 必须与 `sql/` 目录下的文件完全一致。

### 2.2 数据库工具模块 `db.py`

四个函数覆盖所有场景：

| 函数 | 用途 | 返回值 |
|------|------|--------|
| `query(sql, params)` | SELECT 多行 | `list[dict]` |
| `query_one(sql, params)` | SELECT 单行 / COUNT | `dict \| None` |
| `execute(sql, params, conn?)` | INSERT / UPDATE / DELETE | `int`（lastrowid 或 rowcount） |
| `transaction()` | 事务上下文管理器 | `conn`（传给 execute 的 conn 参数） |

**设计理由**：
- `DictCursor` 使查询结果直接可 JSON 序列化，无需转换
- `execute` 支持可选的 `conn` 参数：不传则自动提交，传入则加入事务
- 事务用 `with transaction() as conn:` 语法，退出时自动提交，异常自动回滚

### 2.3 不使用 ORM

保持 SQL 可见性，与课程教学内容直接对应。所有 SQL 逻辑在 `sql/` 目录和路由代码中清晰可读。

## 3. 编码规范

### 3.1 SQL 占位符

**全部使用 `%s`**（PyMySQL 风格），不使用 `?`。参数传元组。

```python
# 正确
query("SELECT * FROM patient WHERE case_no = %s", (case_no,))

# 错误
query("SELECT * FROM patient WHERE case_no = ?", (case_no,))
```

### 3.2 单参数元组

单参数必须带逗号 `(val,)` 构成元组，否则 PyMySQL 会将字符串逐字符拆解。

```python
# 正确
query_one("SELECT * FROM patient WHERE case_no = %s", (case_no,))

# 错误
query_one("SELECT * FROM patient WHERE case_no = %s", (case_no))   # 少了逗号
```

### 3.3 HTTP 状态码约定

| 状态码 | 场景 |
|--------|------|
| 200 | 查询成功、更新成功 |
| 201 | 创建成功（POST） |
| 400 | 业务规则拒绝（余额不足、状态不允许、删除前置检查不通过） |
| 401 | 登录失败（账号不存在、密码错误） |
| 404 | 资源不存在 |
| 409 | 冲突（排班时间段重叠、诊室被占用、重复出院） |

### 3.4 响应格式

- **查询类**：直接返回 `list[dict]` 或 `dict`，FastAPI 自动 JSON 序列化
- **操作类**：返回 `MessageResponse`（含 `message` + `affected_rows`）
- **创建类**：返回 `{"xxx_id": id, "message": "..."}`（含新记录的 ID）

### 3.5 路由函数结构

每个函数遵循以下顺序：
1. 参数校验（FastAPI + Pydantic 自动）
2. 业务校验（冲突检测、前置检查）→ 失败抛 HTTPException
3. 数据库操作（单条 execute 或事务 transaction）
4. 返回结果

```python
@router.post("", status_code=201, summary="简短中文摘要")
def create_xxx(data: SomeSchema):
    """对应 SQL x.x — 业务描述"""
    # 1. 业务校验
    conflict = query_one("SELECT COUNT(*) AS cnt FROM ... WHERE ...", (...))
    if conflict["cnt"] > 0:
        raise HTTPException(409, "冲突描述")

    # 2. 数据库操作
    new_id = execute("INSERT INTO ... VALUES (...)", (...))

    # 3. 返回
    return {"xxx_id": new_id, "message": "创建成功"}
```

### 3.6 事务写法

```python
with transaction() as conn:
    id_a = execute("INSERT ...", params1, conn=conn)
    execute("UPDATE ...", params2, conn=conn)
    # 任何一步失败 → 全部回滚
# with 块正常退出 → 自动提交
```

**注意**：事务内的 SELECT 也需要 FOR UPDATE 锁定行，防止并发问题（见 M7、M8、M9）。

### M3 排班管理
- `m3_schedule.py`：排班 CRUD + 两层冲突检测
- 冲突检测公式：`A.start_time < B.end_time AND A.end_time > B.start_time`
- R03：同一医生同日时间段不重叠
- R04：同一诊室同时段不冲突（仅门诊坐诊）
- R05：门诊坐诊必须指定 clinic，住院巡诊 clinic 必须为空（由 Pydantic model_validator 实现）
- 编辑时排除自身：`AND schedule_id != %s`

### M4 挂号
- `m4_registration.py`：查询可选医生 → 判定初诊/复诊 → 创建挂号 → 取消挂号
- 初诊/复诊由后端判定（查询历史已就诊记录），前端仅展示结果
- 取消仅限"待就诊"状态

### M5 门诊接诊与处方
- `m5_outpatient.py`：接诊队列 → 接诊 → 开具处方（事务）→ 查看处方明细
- 事务三步：创建 outpatient_prescription → 逐条 prescription_detail → 可选扣库存
- 处方费用 = 医生诊疗费 + Σ(药品数量 × 单价)
- 库存扣减检查：`WHERE stock_qty >= %s`

### M6 缴费与历史
- `m6_payment.py`：待支付列表 → 费用汇总 → 支付 → 门诊/住院历史 → 住院费用明细
- 支付乐观锁：`WHERE pay_status = '待支付'`，affected_rows=0 表示已支付
- R31：门诊和住院历史分开查询，住院费用明细通过 4 表 JOIN

### M7 住院登记
- `m7_admission.py`：可选病房 → 可选床位 → 入院（事务）
- 事务五步：FOR UPDATE 锁床位 → INSERT admission_file → UPDATE patient → UPDATE bed → UPDATE ward
- FOR UPDATE 防止并发分配同一床位

### M8 每日查房
- `m8_daily_rounds.py`：我的在院病人 → 查房（8 步事务）→ 历史记录
- R21：同档案同日不可重复查房
- 当日费用 = 床位费（charge_rate）+ 药费（Σ qty × unit_price）
- Step H 扣减预交金使用 `WHERE deposit_balance >= daily_fee`，不足时 affected_rows=0 → 回滚

### M9 出院结算
- `m9_discharge.py`：出院（7 步事务）→ 费用汇总
- FOR UPDATE 锁档案 → 校验未出院 → 校验出院时间 → 汇总费用 → 结算 → 填写出院时间 → 释放资源
- 结算金额 = 预交金 - 总费用（正退负补）
- R25：出院时间必须晚于入院时间
- R28：释放病人状态、床位、病房占用数

### 统计查询
- `stats.py`：医生工作量（R30）、排班、科室、药品、床位使用率
- 全部 GET 请求，时间段参数通过 query string 传入

## 5. 后续开发约束

### 5.1 绝对规则

1. **所有 SQL 必须在 `sql/` 目录有对应文件**，路由中的 SQL 与 SQL 文件保持一致。修改 SQL 时两边同步更新。
2. **所有数据库访问通过 `db.py`**，不得在路由中直接 `import pymysql` 或创建连接。
3. **所有请求体使用 Pydantic Schema 校验**，不得在路由函数内手动校验参数。
4. **所有写操作注意并发安全**：涉及余额、库存、床位分配的 UPDATE 必须带 WHERE 条件（乐观锁）或 FOR UPDATE（悲观锁）。
5. **事务操作必须用 `with transaction() as conn:`**，不得手动管理 commit/rollback。
6. **错误信息用中文**，与前端展示一致。

### 5.2 约定

- 路由函数命名：`list_xxx`（列表）、`create_xxx`（创建）、`update_xxx`（更新）、`delete_xxx`（删除）
- 路由路径用复数名词：`/api/schedules`、`/api/registrations`
- 子资源用路径参数：`/api/schedules/{schedule_id}`
- Swagger summary 用中文简短描述

### 5.3 已知待办（TODO）

### 5.4 禁止事项

- 不要在路由中直接拼接 SQL 字符串（防 SQL 注入），始终使用 `%s` 参数化
- 不要在事务外执行多表关联写操作
- 不要跳过 Pydantic 校验直接操作数据库
- 不要在生产环境使用 `allow_origins=["*"]`（CORS 当前为开发便利全放通）

## 6. 前端技术栈与架构

**日期**：2026-05-24

### 6.1 技术选型

| 维度 | 选择 | 理由 |
|------|------|------|
| 架构模式 | SPA (Single Page Application) | 无页面刷新、流畅体验、无需构建工具 |
| UI 框架 | Bootstrap 5.3 CDN | 响应式组件、中文友好、零配置 |
| 路由 | 手写 Hash 路由器 (~50行) | 轻量、支持 `:param` 占位符、15 条路由够用 |
| 状态管理 | localStorage | 匹配当前无 JWT 的 API，存储登录用户信息 |
| API 通信 | fetch + 统一封装 `api.js` | GET/POST/PUT/DELETE 四方法，自动 JSON 解析，中文错误提示 |

### 6.2 文件结构

```
static/
├── index.html              # SPA 入口（Bootstrap CDN + 脚本加载）
├── css/style.css           # 全局样式（登录页渐变、侧边栏、统计卡片、步骤条）
└── js/
    ├── api.js              # fetch 封装
    ├── auth.js             # getUser / saveUser / logout / isLoggedIn
    ├── router.js           # Router 类（on / go / start / fallback）
    ├── render-utils.js     # showToast / renderTable / mountTable / createModal / escapeHtml
```

### 6.3 路由守卫机制

所有需要登录的页面用 `requireAuth(handler)` 包装：
- 检查 `localStorage` 中是否有用户信息
- 未登录 → 自动跳转 `#/login`
- 已登录访问登录页 → 自动跳转对应角色首页

### 6.4 前后端字段映射

开发过程中发现并修复了以下关键映射问题：

| 问题 | 影响 | 修复 |
|------|------|------|
| 登录接口不返回 `name` | 前端导航栏用户名显示为 "undefined" | 后端 `/auth/login` 增加名字查表逻辑 |
| 登录页提示 `10001/20001` | 种子数据实际 ID 为 1-5 / 1-3 | 更新登录页测试账号提示 |
| 处方创建不返回费用 | 医生开方后看不到费用明细 | `/outpatient/prescriptions` 返回 consultation_fee + drug_fee + total_fee |
| 查房病人列表不返回 charge_rate | 查房表单床位费显示为 50（fallback） | `/rounds/my-inpatients` JOIN ward.charge_rate |
| 住院历史不返回 file_no | 查看费用明细时无档案号参数 | `/payment/history/inpatient` SELECT 增加 af.file_no |
| 后端字段 `name` vs 前端 `doctor_name` | 表格列空白 | 前端全局统一使用后端真实字段名（name, med_name, ward_no, reg_date 等） |
| 入院缺少医生选择 | 管理员入院时 ref_id=999 不是有效医生 ID | 入院表单第 3 步增加医生下拉框 |

### 6.5 已知差异（设计上的简化）

- **药品名称**：数据库字段为 `med_name`，后端返回 `med_name`，前端展示列标题为"药品名称"
- **病房字段**：数据库为 `ward_no`（如 "W-301"），无独立名称字段
- **科室无 description**：后端只返回 `dept_name, dept_type, phone`，前端用 `dept_type` 展示
- **门诊历史**：后端返回 `time`（挂号日期）而非 `reg_time`，字段名略有不同
- **住院费用明细**：后端返回每条药品明细（非按日聚合），前端以平铺表格展示
